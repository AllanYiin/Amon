# Output template

這份模板定義最終回覆的骨架。可以依任務內容增減小節，但不能拿掉必備交付物與必備章節。

## Minimum depth rules

- 這不是大綱模板，而是完成稿模板。
- 每個章節都要填入實際內容，不可只留標題。
- 每個主要章節至少要有：
  - 2 句以上說明，或
  - 1 個有實際內容的表格，或
  - 1 組具體規則 / 流程 / 狀態 / 欄位定義
- 若某節以表格呈現，表格內必須填具體內容，不可只留欄位名。

在正式 spec 前，先輸出研究與比較結果。預設停在這裡等使用者確認；只有在使用者明確要求單回合完成時，才在同一回合繼續。

技術版硬規則：
- 一定要定義專案目錄規劃，且要和模組邊界一致。
- 一定要定義 state 與持久化。
- 一定要定義核心資料模型與資料生命週期。
- 一定要定義通知與背景執行的觸發、狀態、取消、重試與失敗處理。
- 一定要定義錯誤處理、回退策略與可觀測性。
- 一定要先定調 UI 風格，再定義色彩策略。
- 一定要說明是否使用 tabs、wizard、step navigation 或同頁分段。
- 一定要保留單一明確視覺重點，不可把所有區塊都做成同等強度。
- 一定要定義 UI 狀態保存與重新開始機制。
- 任何新增物件都要有 Update / Delete。
- 任何上傳都要有預覽。
- 任何縮放都要維持原始寬高比。
- 任何 LLM 輸出都要明寫 Streaming。
- 一定要列 UI 元件清單、UI 事件回報與 UI ↔ API Mapping。

Stage 計畫硬規則：
- Stage 數量依規格切分，不要硬塞成固定 4-6 階段。
- 每個 Stage 都必須可交付、可測試、可回滾。
- 若涉及 LLM，Instructions 要明寫 Streaming。
- Instructions 要自足，不要假設 Codex 會自動懂上下文。
- 每個 Stage 都要寫實際內容，不能只放欄位名或一句空話。

## SVG minimum rules

- 使用 `viewBox`。
- 使用淺色背景與高對比文字。
- 標籤以繁體中文為主。
- 架構圖要看得出模組邊界與資料流。
- UI 圖要看得出控制項位置、資訊區塊、主要操作動線。

## Interaction rules

- 預設先做研究 checkpoint，再等使用者確認。
- 若使用者明確要求單回合完成，也不能跳過研究 checkpoint。
- 中途可見內容一律用 code block 呈現，不要只在內部思考。

## Project directory rules

- 技術版要有獨立的「專案目錄規劃」章節。
- 目錄規劃至少包含：
  - 根目錄主要資料夾
  - 每個資料夾的責任
  - 代表檔案或進入點
  - 命名原則
  - 測試、資源、設定、文件放置位置
- 若是 monorepo、多套件、前後端分離或多服務架構，要明確說明切分方式。

## UI planning rules

- 先回答介面風格走向，再做顏色與控制項選擇。
- 主色/輔色/強調色要有明確責任，不要隨意混用。
- 若任務具有明顯階段性，優先拆步驟，不要把所有操作塞在同一頁。
- 每個畫面只保留一個主要視覺重點。
- 優先讓主要操作在單一可視畫面內完成。
- 沒有明確用途的控制項不要顯示。
- 要有狀態保存，也要有重新開始。
- 除非使用者指定，預設淺色模式。
