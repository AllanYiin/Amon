# Plain-language rules

白話版規格的受眾是「沒有開發經驗者」。原則不是把技術詞加括號解釋，而是整句改寫。

## 核心規則

- 先寫使用者能做什麼，再寫系統會怎麼回應。
- 先寫畫面、按鈕、清單、提示語，再寫資料與規則。
- 若一句話必須提到內部機制，改成使用者能感受到的結果。
- 若不確定某個詞算不算技術詞，傾向改寫成更生活化的說法。

## 常見禁語與替代表述

| 避免使用 | 改寫方向 |
|---|---|
| API | 跟其他服務交換資料的方式 |
| DB / Database / 資料庫 | 系統記住內容的地方 |
| Backend / 後端 | 系統在背景處理事情的部分 |
| Frontend / 前端 | 你眼前操作的畫面 |
| Schema | 欄位規則 |
| State machine | 流程狀態變化規則 |
| Endpoint | 某個特定功能的連接位置 |
| Migration | 資料升級步驟 |
| Cache | 暫時加速用的記錄 |
| Queue | 排隊等待處理的工作 |
| Cron / Batch job | 定時執行的背景工作 |
| Streaming | 邊產生邊顯示 |
| Token | 內容長度成本 |
| CRUD | 新增、修改、刪除 |
| Auth / Authorization | 登入與權限判斷 |
| WebSocket | 即時連線通道 |
| Model / LLM | 生成內容的智慧引擎 |

## 推薦句型

- 「你建立一個新專案後，下次再打開還能從上次進度繼續。」
- 「上傳檔案後，畫面會先顯示內容預覽，讓你確認是不是選對檔案。」
- 「當內容正在生成時，結果會一段一段出現在畫面上，不需要整份等完。」
- 「放大或縮小圖片時，畫面會保持原本比例，不會被拉扁或拉長。」

## 自我檢查

- 有沒有一句話離開工程背景就看不懂？
- 有沒有把內部做法當成使用者應該知道的前提？
- 有沒有為了偷懶，直接把技術版用同義詞替換？

若答案是有，重寫，不要硬交。
