# Chinese naturalization notes

這份文件整理 `humanize-text` 在繁體中文與中英混寫情境下應優先採用的改寫原則。這不是翻譯規範大全，而是把外部參考收斂成對「自然化」最有用的可執行規則。

## Source takeaways

### From referenced projects
- `rebel47/ai-text-humanizer`
  - 重點在把僵硬文字重新改寫，而不是只做表面同義詞替換。
  - UI 與產品定位偏向一鍵改寫，但沒有提供繁中寫作規範，因此中文策略需要另外補。
- `DadaNanjesha/AI-Text-Humanizer-App`
  - 內容處理偏向風格轉換與正式化，可借鏡「先定輸入，再做風格改寫」的流程。
  - 這類設計若直接套到中文，很容易把文字改得太書面、太像翻譯稿，因此繁中要額外收緊。
- `humanizeai.pro`
  - 市場訴求包含把文字改得更像人寫，但其商業敘事混入偵測器規避話術。
  - 本 skill 只吸收「提高自然度與可讀性」這個正當目標，不吸收規避偵測定位。
- `dustin/go-humanize`
  - 這是數字、時間、位元組的人類可讀格式化函式庫，不屬於本文字 naturalization 範圍。
  - 這個參考最重要的用途是提醒 skill 邊界，避免 `humanize` 一詞過寬而誤觸發。

### From Chinese writing guidance
- Mozilla Localization Style Guide zh-TW
  - 強調在地化語感與一致性，提醒不要硬翻專有名詞，也不要讓文字失去在地讀感。
- Gengo Chinese Style Guide
  - 強調避免直譯、避免不自然的被動結構、維持可讀性與語境一致。
- 華梵大學中文寫作教材
  - 提醒段落應有連貫與中心，避免碎裂成一句一段的拼貼式寫法。
- `語言模型書寫風格分析.pdf` 與新增文章
  - 補上「機器感」更細的診斷訊號：過度平均、教學式骨架、轉折詞濫用、在地語感不對、句長與標點節奏太整齊。
  - 補上可操作修法：style spec、few-shot、反例清單、去模板、去重複、節奏化與文化提示。

## Rewrite priorities for zh-TW

1. 先順語序，再談華麗修辭。若原句保留英文邏輯順序，中文通常會顯得卡。
2. 優先把抽象名詞改成動作與主體，例如把「進行優化」改成「優化」，把「提供支援」改成「支援」。
3. 避免被動句堆疊。英文常用的被動句搬到中文，通常會變得生硬。
4. 保留產品名、品牌名、技術名詞的官方寫法，不擅自創造不常見翻譯。
5. 中英混寫時，只要同一份文字決定某個詞保留英文，就要前後一致。
6. 段落節奏要有變化，不要每句都一樣長，也不要每段都用同一種開頭。
7. 不要預設第一段一定先下定義，也不要預設最後一定做總結；人類文章常直接從觀點、場景或問題切入。
8. 先讓讀者看懂，再讓文風變漂亮。不要為了「人味」硬加感嘆、俚語或虛構細節。
9. 允許輕微的不平均。人類寫作常有重點段落較長、轉折段較短，這種落差是正常訊號。

## Common anti-patterns

### Translationese
- 特徵：
  - 句子過長
  - 修飾語一層層堆在名詞前
  - 過度保留英文轉折詞與抽象詞
- 修法：
  - 拆句
  - 把重點主詞提前
  - 改成主詞加動作的句式

### Template voice
- 特徵：
  - 每段都像同一個 prompt 生出來的
  - 常見句型連續重複
  - 高頻空話，例如「全面提升」「高效賦能」
- 修法：
  - 刪掉空泛形容詞
  - 改用具體結果或具體動作
  - 打散過度對稱句型

### Explain-first scaffold
- 特徵：
  - 第一段先下定義
  - 中段照著教學順序均分
  - 最後硬補一段總結
  - 過度使用「首先、其次、最後、總而言之、以下是」
- 修法：
  - 若語境允許，直接從觀點、例子或場景切入
  - 把不必要的轉場拿掉
  - 只保留真的能幫助理解的承接，不保留講義骨架

### List residue
- 特徵：
  - 原文雖改成段落，但仍可看出每句對應一個 bullet
  - 大量使用「首先、其次、再次、最後」卻沒有真正的段落流動
- 修法：
  - 先判斷哪些項目能併成同一意圖
  - 用轉承關係串成自然敘述
  - 只保留必要的過渡語，不保留列表節奏

## Paragraph reconstruction pattern

當輸入是列點、提綱或摘要時，請依這個順序重組：

1. 先辨識每條列點是在說背景、功能、結果還是限制。
2. 把同類資訊合併，不要一條列點對應一個句子。
3. 為每段指定單一重點，常見是：
   - 問題與背景
   - 做法與差異
   - 結果與限制
4. 段與段之間用自然承接，不用列表標記。

## Style anchoring with samples

若使用者提供一小段他喜歡的文字，不要照抄字句；請只學以下幾件事：

1. 句長分布是偏短、偏中、還是長短交錯。
2. 段落是直接陳述、先講故事、還是先丟觀點。
3. 語氣距離是親近、冷靜、正式、半口語。
4. 細節密度是多例子、多觀察，還是偏抽象評論。

這種做法比要求模型「像某某作家」更穩，也比較不會變成抄襲。

## Safe phrasing policy

- 不聲稱保證通過 AI detector。
- 不補寫「我實際用過」「我親自測過」這類未提供的第一手經驗。
- 若原文事實不完整，只能保守保留，不要擅自補全。

## Practical checklist

- 讀起來是否像繁中原生寫法，而不是英文照譯。
- 段落是否連貫，而不是 bullet points 失去符號後的殘影。
- 專有名詞是否一致。
- 是否新增任何原文沒有的事實、情緒或經驗。
- 是否仍可清楚看出受眾與場景。
