# Output contract

`humanize-text` 的預設輸出是完成稿，不是分析報告。除非使用者明確要求比較版、修改說明或逐段註解，否則直接交付可用文字。

## Default output shape

- 只輸出完成稿。
- 以自然段落為主，可有短標題，但不得有條列式或有序列表。
- 若原文資訊量較大，優先拆成 2-4 段，不要塞成單一超長段。
- 不主動附「以下是優化後版本」「修改如下」這種模板式前言，除非使用者要比較。

## Hard constraints

- 不新增原文沒有的事實。
- 不假裝真實個人經驗。
- 不保證通過 AI detector。
- 不輸出 `-`、`*`、`+`、`1.`、`1)`、`一、`、`二、` 這類列表形式。
- 不用「以下是」「總而言之」「首先／其次／最後」當成預設骨架，除非原文語境確實需要。

## When explanation is requested

若使用者明確要解釋修改重點，先給完成稿，再用短段落說明，不用條列。
若非必要，不要回到「列三點原因」這種機器化說法。

## Self-check before sending

- 讀一遍，確認段落之間真的有轉承。
- 掃描行首，確認沒有列表標記。
- 對照原文，確認沒有新增觀點或缺漏關鍵名詞。
- 若是繁中，確認讀感不是中國用語直套或英文語序硬譯。
- 確認第一段不是在做沒必要的定義式開場，最後一段也不是硬補總結。
