# Query Playbook

這份文件把 `web_tools.py` 的搜尋經驗整理成可重用規則，並補上較穩定的安全邊界。

## 1. 先想頁面，不要先想關鍵字

查詢前先回答：答案最可能在哪類頁面？

- 法規 / 政府公告
- 公司 investor relations / annual report
- GitHub repo / 官方文件
- PDF / 白皮書 / 規格
- 新聞稿 / 媒體採訪
- 開放資料頁

這一步對應 `better_keywords` 的核心：用第一性原則推回查詢，而不是直接把使用者原句丟出去。

## 2. 查詢族，而不是單一長句

預設做 2-5 組查詢，每組只服務一個假設。

好模式：
- `"Model Context Protocol" specification`
- `"Model Context Protocol" site:modelcontextprotocol.io`
- `"Model Context Protocol" GitHub`
- `company name "annual report" filetype:pdf`

壞模式：
- `我想找 mcp protocol 官方規格以及這兩年的教學跟實作案例`

## 3. 每組保留 2-3 個核心概念

對應 `better_keywords` 的做法：
- 1 個主概念
- 1 個來源或頁面限定詞
- 視需要再加 1 個年份 / 地區 / 版本詞

若超過 3 個主要概念，先拆成兩條查詢。

## 4. 先用通稱壓縮，再必要時拆批

當使用者一次塞很多相關詞時，先找上位概念壓縮：
- `損益表` 取代 `營收 毛利 營業利益`
- `investor relations` 取代 `annual report earnings presentation filings`

若壓縮後仍然太寬，再拆成多組查詢。

## 5. 操作符與穩定性

優先使用目前普遍穩定的技巧：
- `site:`：限定網站或網域
- 引號：固定短語、法規名、公司名、產品名
- 排除詞：去掉錯誤語境
- `filetype:`：找 PDF、XLSX 等檔案
- 進階搜尋或日期工具：控制時間、語言、區域

護欄：
- `before:` / `after:` 可當作經驗性捷徑，但不要視為所有搜尋引擎都穩定支持的正式規則。
- 若時間篩選很重要，優先用搜尋工具、進階搜尋頁或站內日期頁面。

## 6. 引擎切換要有理由

不要隨機切換引擎。合理理由包括：
- 第一引擎結果被 SEO 頁洗掉
- 需要另一個排序觀點做交叉驗證
- 某引擎對特定網站收錄較差

## 7. 控制每輪打開的結果量

建議上限：
- `answer`: 每組先看前 3 個
- `gathering`: 每組先看前 5 個
- `full-list`: 每組最多 5-8 個，再去重

不要看到搜尋結果就全部打開；先從網域、標題、摘要過濾。

## 8. 第二輪搜尋要針對缺口

只在下列情況進第二輪：
- 還沒找到官方來源
- 找到的是二手轉述
- 缺年份、版本、地區
- 找到的是頁面，但真正需要的是 PDF / 原始文件

第二輪應修改查詢假設，而不是單純把原查詢加長。

## 9. 常見反模式

- 直接搜尋整段自然語言
- 為了精確而堆過多 operator
- 同時加兩個 `site:` 卻沒理由
- 拿未查證的 operator 當標準答案教給使用者
- 找到第一篇文章就停止，不做來源交叉驗證
