# Framework Selection

這份文件用來避免「先拿到框架，再反推問題」的常見錯誤。

## 先判斷問題性質

### 1. 可遞迴、可分治、合併規則明確
- 主工具：Divide and Conquer
- 適用：演算法、資料處理、可重複模組化流程
- 不足：通常不足以處理跨部門、價值權衡、驗收與資源衝突

### 2. 交付或範疇要說清楚
- 主工具：WBS
- 適用：專案、產品功能、跨團隊交付、明確產出物
- 目標：把全部授權範疇拆成可管理工作包
- 常見錯誤：拿 WBS 當分類法，導致層級漂亮但無法交付

### 3. 需要分類、分析、決策架構
- 主工具：MECE / Issue Tree
- 適用：策略分析、問題診斷、論點架構、報告整理
- 目標：互斥且窮盡，避免分析重疊或遺漏
- 常見錯誤：以為 MECE 就等於排程或交付分解
- 使用提醒：
  - 先定義根問題句，再拆第一層
  - 同一層只用一種分類基準
  - 原因樹、對策樹、WBS 要分開處理

### 4. 問題會反覆發生，且存在副作用、延遲、互相影響
- 主工具：系統思考 / 回饋迴路
- 適用：流程瓶頸、組織摩擦、供需波動、留存/轉化、跨部門協作
- 目標：先找系統如何產生目前行為，再找槓桿點
- 常見錯誤：只列待辦，沒有指出哪個結構在持續製造問題

### 5. 成功標準不清，需要先對齊結果
- 主工具：Objective / KR / 里程碑
- 適用：跨角色對齊、季度計畫、研究題、變革專案
- 目標：把問題改寫成可觀察的終點狀態
- 常見錯誤：KR 寫成活動而不是結果

### 6. 高不確定、研究或創新題
- 主工具：Baseline + PoC + PERT/三點估算 + Go/No-Go
- 適用：研究驗證、新技術導入、探索型專案
- 目標：先降低不確定性，再擴大投入
- 常見錯誤：一開始就排完整路線圖，卻沒有基準線與決策點

## 常見組合

- WBS + CPM/PERT：當你既要拆工作，又要處理依賴與工期
- WBS + DoD/AC：當你怕任務拆出來卻沒法驗收
- 系統思考 + Kanban：當你要一邊看結構、一邊控流量與瓶頸
- Objective/KR + WBS：當你要先對齊結果，再落到交付
- Baseline + PoC + PDCA：當你要在高不確定中逐步學習

## 快速判斷句

- 如果你在問「怎麼分類才完整」：先想 MECE
- 如果你在問「真正問題是什麼、表象背後卡在哪」：先想 issue tree
- 如果你在問「要交什麼、誰做什麼」：先想 WBS
- 如果你在問「為什麼老是復發」：先想系統思考
- 如果你在問「到底算成功了沒」：先想 Objective/KR
- 如果你在問「這個想法值不值得做」：先想 Baseline、PoC、決策門檻

## 額外判斷提醒

- 問題先寫成「現況與期望的落差」，不要直接把觀察到的表象當根因。
- 如果你還在找原因，先做原因樹；如果你已經知道方向要落地，才轉 WBS。
- 若問題只在某些客群、時間、地區或流程節點發生，先用層別法決定第一層切法。

## 最低品質門檻

- 框架選擇必須附理由
- 至少說清楚主框架與輔助框架
- 不能把框架名稱當答案本身
- 不能把症狀、根因、對策混寫在同一層
