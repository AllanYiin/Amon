# Problem Framing And Issue Trees

這份文件補強「問題拆解」最常失真的地方：還沒定義真正問題，就急著做工作分解。

## 先把問題寫成落差

優先用這四格整理，而不是直接列解法：

| 欄位 | 要回答什麼 |
|---|---|
| 現象 | 你現在看到什麼不對勁 |
| 期望狀態 | 本來應該長什麼樣 |
| 差距 | 目前和期望差在哪裡、差多少 |
| 根因假設 | 目前懷疑是哪些原因造成，哪些還沒證實 |

如果只有現象，先承認只知道現象，不要假裝已經知道根因。

## 先決定要畫哪一種樹

- `原因樹 / issue tree`：用來拆「為什麼會這樣」，重點是分類完整與假設驗證。
- `對策樹`：用來拆「可以怎麼解」，重點是短期止血、長期改善與副作用取捨。
- `WBS`：用來拆「要交什麼、誰做什麼」，重點是交付、依賴與驗收。

不要把這三種樹混在一起。若還在找原因，就不要把未驗證猜測直接寫成任務。

## Issue Tree 的最低規則

1. 根問題句先寫清楚。
2. 同一層分支必須回答同一個問題。
3. 同一層只用一種分類基準。
4. 分支應盡量互斥且完整；若做不到完整，直接標示目前缺口。
5. 每個分支都要能往下接「證據怎麼驗證」或「下一步如何收斂」。

## 分類基準怎麼選

若不知道第一層怎麼拆，先選一種最有辨識力的層別方式：

- 客群
- 流程階段
- 產品/服務線
- 通路
- 地區
- 時間
- 原因類別

不要在同一層同時混用「客群 + 原因 + 時間」。

## 何時用差異分析、層別法、5 Why

- `差異分析`：拿「好的樣本 vs 壞的樣本」比較，適合先找差異從哪個面向冒出來。
- `層別法`：當問題只在某些客群、地區、時段、流程節點出現時，用來決定第一層切法。
- `5 Why`：當單一路徑因果明顯時，用來往下深挖；若問題本身是多因子，不要只靠 5 Why。

這三者是收斂工具，不是 WBS 的替代品。

## 對策設計要分短期與長期

當問題已明確存在營運損失或服務風險時，對策至少分兩層：

- 短期止血：先控制損失、降低波動、避免繼續惡化。
- 長期改善：改掉真正製造問題的結構、流程或機制。

若只有短期止血，通常會復發；若只講長期理想，通常來不及救火。

## 從分析轉執行的轉換點

只有在主要假設已有足夠證據時，才把分支轉成工作包。轉換時至少補：

- 要驗證或改變哪件事
- 預期輸出
- 驗收方式
- 依賴
- 若失敗要怎麼回退或改判斷
